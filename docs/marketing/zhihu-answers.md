## 1. "使用 ChatGPT 时如何保护个人隐私？"

**问题链接：** 搜索 "使用 ChatGPT 时如何保护个人隐私" 相关问题

**回答草案：**

先说结论：**你发给 ChatGPT 的每一条消息，OpenAI 都能看到。** 这不是阴谋论——API 服务端必然需要读取你的输入才能生成回复。

具体来看三个层面的隐私风险：

**1. 数据存储**
OpenAI 的 API 数据保留政策一直在变。目前的 API 政策是不使用 API 数据训练模型（Enterprise 和 API），但：
- 数据仍然经过 OpenAI 服务器
- 错误日志可能包含你的输入内容
- 你无法独立验证数据是否被删除

ChatGPT 免费版/Plus 版的对话数据默认会被用于模型训练（除非在设置中主动关闭）。

**2. 你发了什么**
大多数用户意识不到自己在泄漏什么。我见过真实案例：
- 开发者把整个 `.env` 文件贴进 ChatGPT 调试 bug
- 客服人员把客户订单详情贴进去写回复邮件
- 医生把患者病历摘要贴进去写诊断建议
- HR 把候选人简历贴进去写评估报告

每一个场景都涉及真实的个人隐私数据。

**3. 怎么办**

最直接的办法：**在数据离开你机器之前脱敏。**

```
你的电脑 → [脱敏代理] → OpenAI 服务器
                ↑
         手机号 → [PHONE_1]
         邮箱   → [EMAIL_1]
         身份证 → [ID_1]
```

具体操作（30 秒）：

```bash
docker run -d -p 9999:9999 ghcr.io/gunxueqiu6/ai-privacy-gateway:lite
```

然后把你的 AI 客户端 API 地址改为 `http://localhost:9999`。

原理很简单：一个本地反向代理，在请求转发给 OpenAI 之前，用正则检测并替换掉所有敏感信息（手机号、邮箱、身份证、银行卡、地址、API 密钥等 14+ 类）。GPT 不需要知道你的真实手机号——它只需要知道"这里有一个手机号"。

**开源（MIT 协议），完全本地运行，不依赖任何外部服务。** 你不能把自己的数据安全寄托在 AI 公司的"承诺"上。

在线演示（浏览器本地处理，不传输数据）：https://privacygw.pages.dev/demo

## 2. "Cursor/Copilot 会把代码上传到服务器吗？"

**回答草案：**

简短回答：**是的，Cursor 和 Copilot 都需要把你的代码发送到服务器才能工作。**

长回答：

Cursor 和 GitHub Copilot 的 AI 功能都依赖云端 LLM API。当你使用以下功能时：

- **Tab 补全 (autocomplete)**：当前文件的部分上下文发送到服务器
- **Ctrl+K 重构**：选中的代码发送到服务器
- **Chat / Composer**：对话内容 + 相关文件上下文发送到服务器
- **Agent 模式 （Cursor）**：完整文件内容发送

Cursor 的隐私政策里写了他们会收集"代码片段和使用数据"，Copilot 的遥测数据包括"编辑操作和代码上下文"。

**真实的隐私风险：**

1. **API 密钥泄露**：最常见的事故。开发者在 config 文件里放了真实 API key，调试时整个文件发给了 AI
2. **业务逻辑暴露**：你把核心算法发给 AI 优化——这代码是公司的商业机密
3. **数据库连接串**：`postgres://user:pass@host/db` 出现在代码中，被发送到 AI 服务器
4. **客户数据**：测试 fixture 里的真实用户数据——名字、邮箱、手机号——一并发出去了

**怎么保护？**

方案一：设置 Cursor/Copilot 的 API Base URL 指向本地隐私代理

```bash
docker run -d -p 9999:9999 ghcr.io/gunxueqiu6/ai-privacy-gateway:lite
# Cursor → Settings → API Key → Base URL → http://localhost:9999
```

代理会自动识别并脱敏 20+ 种 API 密钥格式（OpenAI、AWS、GitHub、Stripe 等），以及所有常见 PII（姓名、手机、邮箱、身份证、银行卡）。密钥被检测到后立刻替换为 `[API_KEY_1]` 这样的占位符——AI 服务商永远看不到你的真实密钥。

方案二（Cursor）：Settings 里配置 `.cursorignore` 排除敏感文件

方案三：使用代码库扫描工具定期检查是否有硬编码密钥

**每种方案覆盖不同的风险层。最好的做法是组合使用。** 本地隐私代理覆盖实时 API 调用；`.cursorignore` 限制文件范围；代码扫描覆盖存量风险。

项目地址：https://github.com/gunxueqiu6/ai-privacy-gateway （MIT 开源）

## 3. "DeepSeek 的数据安全吗？"

**回答草案：**

这个问题的答案是分层的。我先说技术事实，你自行判断。

**技术层面：**

DeepSeek 是中国公司，服务器主要在中国大陆。当你调用 DeepSeek API 时：

1. 你的数据跨境传输到中国服务器
2. 数据处理受中国法律管辖（PIPL 《个人信息保护法》、DSL《数据安全法》、CSL《网络安全法》）
3. 中国 PIPL 在框架上类似 GDPR——要求目的限制、最小必要、安全保护——但政府数据访问机制与欧美不同

**对个人的影响：**

如果你是中国用户，数据就在境内，PIPL 对你提供保护。如果你是海外用户（欧美、东南亚等），你的数据跨境进入中国——你需要评估：
- 你的国家对数据出境的法律要求（GDPR 要求对等保护水平）
- 你发送的内容是否包含他人的个人信息（如果是，你可能是数据控制者）
- DeepSeek 的 API 数据处理条款（与消费者聊天版不同，需要仔细阅读）

**对企业和组织的影响：**

这是最大的风险区。企业使用 DeepSeek 可能触发：
- GDPR 跨境数据传输合规要求
- 行业监管（金融：PCI DSS，医疗：HIPAA，上市：SOX）
- 企业内部数据治理政策

**最安全的做法：数据不出去。**

无论你用什么 AI API——中国的 DeepSeek 还是美国的 OpenAI——最安全的策略是：**敏感数据在离开你的环境之前脱敏。**

一个本地代理：

```bash
docker run -d -p 9999:9999 ghcr.io/gunxueqiu6/ai-privacy-gateway:lite

# DeepSeek 客户端配置 API Base URL 为 http://localhost:9999
```

你的真实数据（姓名、手机、邮箱、身份证、银行卡、API 密钥）被替换为类型化占位符后，才离开你的机器。DeepSeek（或任何 AI 服务商）永远看不到原始敏感数据。

把 AI 模型看作一个工具——你给它足够的上下文理解任务，但不需要给它真实的个人信息。

开源工具：https://github.com/gunxueqiu6/ai-privacy-gateway
在线演示：https://privacygw.pages.dev/demo

## 4. "企业使用大模型如何做到合规？"

**回答草案：**

企业使用大模型的合规挑战不是"能不能用"，而是"怎么控制着用"。我拆成四个层面来讲。

**第一层：数据分类**

不是所有数据都不能发给 AI。先分类：
- **禁止类**：个人身份信息、受保护健康信息、支付卡号、商业机密
- **受限类**：内部项目文档、非公开财务数据、客户沟通草稿
- **允许类**：公开文档、通用技术问题、已脱敏数据

禁止类数据必须脱敏后才能发出。

**第二层：技术控制**

不要让"靠员工自觉"成为你的安全策略。技术控制才是可审计的：

```bash
# 1. 部署本地隐私代理（30 秒）
docker run -d -p 9999:9999 ghcr.io/gunxueqiu6/ai-privacy-gateway:lite

# 2. 网络策略：AI 流量强制走代理
# 防火墙规则：只允许代理的出站 AI API 请求

# 3. 所有员工 AI 客户端配置指向代理
# Cursor/VS Code: API Base URL → http://gateway.internal:9999
# ChatGPT 客户端: 同上
```

技术控制的优势：
- **自动脱敏**：手机号、身份证、邮箱等 14+ 类 PII 自动检测替换
- **审计日志**：所有 AI API 调用记录（谁、什么时候、发了什么类型的请求）
- **策略执行**：可以按部门/角色设置不同的脱敏规则
- **不依赖员工意识**：技术兜底，不是靠培训PPT

**第三层：合规对齐**

这段主要讲 GDPR/HIPAA/SOC 2 分别怎么对齐，每个框架的要求不同但数据脱敏是共同基础。

**SOC 2**：需要审计轨迹和访问控制。代理提供完整的审计日志 + 基于角色的策略引擎。
**HIPAA**：需要 PHI 脱敏和 BAA。代理在传输前脱敏 PHI，满足 Safe Harbor 去标识化要求。
**GDPR**：需要数据最小化和传输保障。代理实现数据最小化（AI 看不到原始数据）+ 传输脱敏。
**PIPL**：需要最小必要和跨境合规。代理在数据出境前脱敏。

**第四层：组织和流程**

- 建立 AI 使用政策（不是禁止，是规范）
- 员工培训（不是吓唬，是赋能）
- 定期合规审查
- 供应商风险评估（AI 服务商的数据处理条款）

**一句话总结：技术控制做兜底，流程规范做引导，合规框架做对齐。**

企业方案：https://privacygw.pages.dev/enterprise-ai-data-protection

## 5. "AI API 调用时如何脱敏？"

**回答草案：**

AI API 调用的脱敏和其他场景（如数据库存储、日志输出）不太一样——你不能简单地把数据哈希掉，因为 AI 需要理解语义。你要做的是**保语义、去标识**。

**AI API 脱敏的核心原则：**

- 手机号 → `[PHONE_1]`（AI 知道这是手机号，但不知道具体号吗）
- 不是 → 哈希为 `a3f8c9d...`（AI 完全看不懂这是什么）

**三种方案对比：**

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **本地代理** | 反向代理拦截请求，正则检测替换 | 零代码改动，30秒部署，延迟 <1ms | 非结构化文本检测精度低于 ML |
| **SDK 集成** | 在代码中调用脱敏库 | 可定制，可结合业务逻辑 | 需要改代码，每个项目单独集成 |
| **云端 DLP** | API 调用第三方检测服务 | 精度最高，ML+NLP | 需要把数据发给第三方（本末倒置），延迟高，费用高 |

**推荐：本地代理方案。**

大多数团队不需要 ML 级别的精度——结构化 PII（手机号、身份证、邮箱、银行卡、API 密钥）占实际泄漏的 95%+，正则匹配就够了。

实操：

```bash
# 启动代理
docker run -d -p 9999:9999 ghcr.io/gunxueqiu6/ai-privacy-gateway:lite

# 发送请求
curl -X POST http://localhost:9999/api/mask \
  -H "Content-Type: application/json" \
  -d '{"text": "张三，电话13812345678，身份证110101199001011234"}'

# 返回：
# {"masked": "[PER_1]，电话[PHONE_1]，身份证[ID_1]", "mappings": {...}}
```

支持的脱敏类型（14+）：
- 手机号、邮箱、身份证、银行卡、姓名、地址、公司名、
- IP 地址、车牌号、URL、日期、金额、邮编、
- API 密钥（OpenAI、AWS、GitHub 等 20+ 格式）
- 自定义正则

本地演示（浏览器端处理，数据不上传）：https://privacygw.pages.dev/demo

GitHub（MIT 开源）：https://github.com/gunxueqiu6/ai-privacy-gateway
